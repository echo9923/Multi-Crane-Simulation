  # 任务：Module J 的规划与实现

## 背景

- **Module J 的职责（一句话）**：管理仿真主循环与单帧生命周期，冻结 `WorldSnapshot` 作为决策时刻一致性快照，协调全部模块按固定顺序调用，推进仿真时钟并判定 episode 终止条件——它是整个仿真系统的"调度指挥中心"。

- **上游/下游依赖**：
  - **上游依赖（J 读取所有模块的公开接口）**：
    - 模块 A：`ResolvedConfig`（`sim.dt`、`sim.duration_s`、`controller_hz`、`llm_decision_interval_s`、运行模式 `RuntimeMode`、operator 分配）
    - 模块 B：`CraneConfig[]`（塔吊静态参数，用于构造快照）
    - 模块 C：`CraneState[]`（每帧物理状态，用于冻结快照和推进）
    - 模块 D：`Task[]`、`TaskQueue[]`、任务状态机接口（`activate_due_tasks`、`update_after_physics`）
    - 模块 E：天气模块接口（`weather.update(sim_time)`）
    - 模块 F：`Observation` 构造接口（`build_batch`）
    - 模块 G：LLM 操作员决策接口（`decide_parallel` / `OperatorDecisionOrchestrator`）
    - 模块 H：安全审查接口（`apply_batch`）、在线风险评估接口（`compute_online`）
    - 模块 I：低层控制器接口（`compute_batch`）
    - 模块 K：碰撞检测接口（`detect`）
    - 模块 L：记录与导出接口（`record_step`、`record_initial_frame`）
    - 模块 N：WebSocket 推送接口（`broadcast_sim_frame_if_enabled`）
  - **下游被谁调用**：
    - 所有模块：J 是唯一的主循环持有者，按 `0.7.8` 单帧生命周期依次调用各模块
    - 模块 M（后端 API）：通过 J 启动/停止/查询 episode
    - 模块 N（前端）：通过 J 获取 `SimFrame` 推送
    - 模块 O（数据集生成）：通过 J 批量运行多个 episode

- **必须遵守的对外契约**：
  - 总方案 `0.7.2` 核心对象表：`WorldSnapshot` 的权威 owner 是模块 J，消费者 F/G/H/L；`episode_status` 的权威 owner 是模块 J
  - 总方案 `0.7.3` 模块合同：J 读取全部模块公开接口；产出 episode clock、`WorldSnapshot`、`episode_status`；控制调用顺序、时间推进、episode 状态；不在调度器里内嵌模块业务规则；统一把模块失败映射为 episode/task/run 状态
  - 总方案 `0.7.8` 单帧生命周期：J 必须严格按固定顺序执行每帧 15 步（天气 → 任务激活 → 决策判断 → 冻结快照 → 在线风险 → observation → LLM 决策 → 命令解析 → 安全审查 → 命令过期 → 控制器 → 物理积分 → 任务状态机 → 碰撞检测 → 记录推送 → 终止判定）
  - 总方案 `0.7.9` Replay 严格复现合同：replay 不调用 LLM，按历史 `ExecutedCommand` 重放；检查 `resolved_config_hash` 一致性
  - 总方案 `J.2` 关键约束：同一 `decision_time` 的多塔 observation 必须来自同一 `WorldSnapshot`；同一 `decision_time` 的多塔命令收齐后同时应用；online risk 可在决策前用于 R1 observation；offline label 只能在 episode 后生成
  - 总方案 `J.3` 运行模式：`offline_batch`（无前端批量生成）、`offline_replay`（读取 `command_replay.jsonl` 复现）、`interactive_server`（FastAPI + WebSocket）
  - 总方案 `J.4` 验收标准：仿真不因 LLM 响应慢卡死；LLM 短暂延迟保持上一命令；`command_duration_s` 到期后默认 neutral_stop；LLM 超时后默认 neutral_stop；`offline_wait` 模式可暂停等待 LLM；LLM 连续失败超阈值 → `episode_status=llm_failed`；支持实时和加速离线模式；支持无 LLM 批量生成；支持 replay 模式；记录每一帧状态和每次 LLM 决策时间

## 通用约束

- 阶段一、二不写实现代码。
- 每次提交前本地测试必须通过，不要多个任务攒一起提交。
- 任何对边界/接口的偏离，先说明原因再做。
- `WorldSnapshot` 的 Pydantic schema 是唯一事实源，所有字段必须通过 `extra="forbid"` 校验。
- 调度器不得在伪代码中直接写任务、安全、controller 的业务规则，只能调用各模块合同接口。
- 同一 `decision_time` 的多塔 observation 必须来自同一 `WorldSnapshot`。
- 同一 `decision_time` 的多塔命令收齐后同时应用，不得逐塔逐步应用。
- `command_duration_s` 到期后若无新命令，自动进入 neutral_stop。
- LLM 超时后默认 neutral_stop，连续失败超过 `max_consecutive_failures` → `episode_status=llm_failed`。
- `offline_wait` 模式下仿真时间可暂停等待 LLM 返回；`realtime_stale` 模式下超时使用上一命令。
- 碰撞检测触发后 → `episode_status=failed_collision`，终止 episode，不生成碰撞后轨迹。
- NaN/Inf 状态 → `episode_status=failed_invalid_state`。
- replay 模式不调用 LLM，不调用 summary LLM；历史 summary 只作为日志读取。
- replay 必须检查 `resolved_config_hash`、`command_schema_version`、`physics_schema_version` 一致性。
- 调度器必须对规则司机和 LLM 司机使用完全相同的帧循环——不得有两套调度逻辑。
- 每次帧循环的记录（`record_step`）必须在 physics step 之后、终止判定之前执行。
- 事件归属到检测到该事件的 step 后帧。
- `record_initial_frame` 在 `time_s=0` 时执行，记录初始状态。
- `WorldSnapshot` 不得包含未来标签、offline risk label 或 LLM reason。
